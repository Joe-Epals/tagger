version 1.1 release notes

This section outlines the new features, bug fixes, and behaviors of Tagger version 1.1.

Tagger complies with LRMI V1.1 specification.

New Feature: Image Thumbnail

Tagger now allows an image to be uploaded with each resource. This image will be displayed in the inBloom Search application as well as the Learning Registry Index (LRI) Browser application.

Images should be in a square aspect ratio. Tagger will resize the image to the following:

  • 128 pixels x 128 pixels for inBloom Search
  • 64 pixels x 64 pixels for LRI Browser

If the user uploads an image that is different than a square aspect ratio, Tagger will crop it to a square aspect ratio. If no image is provided, Tagger will attempt to display a screenshot of the webpage associated with the resource’s URL. Because this is an automated process, there is no guarantee what the software will capture. Therefore, uploading thumbnails is highly recommended.

New Feature: Data Validation

Version 1.1 of Tagger corrected a number of issues with data validation to ensure that the data correctly conforms to standards where needed, such as time required and created date fields. Otherwise, Tagger should not limit the user’s ability to import any values in the Education fields.

Tagger will also catch some common CSV file errors such as transposed or missing columns in the CSV import file. A new CSV template that needs to be used when importing has been released. This template is available from Tagger, on the Import screen. The goal of this version of Tagger is to provide an application that facilitates tagging while preventing errors caused by data formatting issues. Tagger performs various checks on import and provides error messages for poorly formed data. Then, the data must be corrected before import will be successful.

New Feature: Publishing to the Learning Registry

This version of Tagger publishes resource metadata to the Learning Registry (http://www.learningregistry.org/). The user can publish to the Learning Registry after signing in via the “Sign In with inBloom” link on the upper right corner of the Tagger application page. To do so, the user must register for a developer account with inBloom.

OTHER Field in Education Tab

The OTHER field has been reinstated for each element under the Education tab, excluding theTime Required field. The user can choose item(s) from a drop-down list and/or add free-text items in the Other field. Use commas to separate each item in the Other field.

Working Without Signing In

Tagger can be used to work on any number of resources without signing in with an inBloom account. The user creates new items by clicking on the “New” button or import items using the “Import” button. The CSV import template can be found in the “LRMI Template Guide” link provided in the Import dialog box.

Clearing Work

If the user is not signed in, refreshing the page will clear all existing work.

Using the CSV Template

The user should use Tagger’s CSV template to ensure that the data is organized correctly for import into Tagger. The template can be found here:

http://tagger.inbloom.org/Tagger-V1.2.0-Template.csv

CREATING A CSV TEMPLATE
Or, a CSV template can be created through the following steps:

  1. Create one resource
  2. Uncheck the new item
  3. Export as CSV via the export button.

TAGGER AND EXCEL
If Excel is used to edit CSV files, Excel interprets the age range as a date. Age ranges are two numbers separated by a hyphen, for example 10-12, or 14-16. Excel interprets “10-12” as “Oct-12.”

There are a few solutions to address this Excel problem of interpreting age ranges as dates.

STARTING WITH A BLANK TEMPLATE
If starting with a blank template, the user should set the “Typical Age Range” column to display text instead of dates through the following steps:

  1. Highlight the column by clicking on the letter above “Typical Age Range”.
  2. Choose “Cells...” from the “Format” menu.
  3. Click “Text” in the “Category” list.
  4. Click the “OK” button

EDITING AN EXISTING CSV FILE
There are two ways to solve this problem when editing an existing CSV file.

WORKFLOW #1
Instead of opening the CSV file with Excel by double clicking it, the user can use the File/Import feature to import the CSV file into Excel:

  1. Do not click on the file to open it, or do not open it with Excel directly. Instead, within Excel, go to the File Menu and select Import
  2. Select File Type = CSV, and then navigate to the CSV file that needs to be imported
  3. Select Delimited and not Fixed Width
  4. The import wizard allows the user to set the Data Format for each column. Go to Typical Age Range and set the column Data Format to text.

WORKFLOW #2
If the user opens an existing CSV file with data in Excel directly, Excel will interpret the following age ranges as dates:

  • 3-5 as 5-Mar
  • 5-8 as 8-May
  • 10-12 as 12-Oct
  • 12-14 as 14-Dec

Changing the column to display as text will change the dates to the numbers Excel uses internally for dates:

  • 3-5 becomes 41338
  • 5-8 becomes 41402
  • 10-12 becomes 41559
  • 12-14 becomes 41622

To make Excel display the correct age ranges, the user must search and replace on each number, but add a leading apostrophe to the age range to force Excel to interpret it as a string.

  • 3-5 becomes 41338. Find 41338 and replace with ‘3-5
  • 5-8 becomes 41402. Find 41402 and replace with ‘5-8
  • 10-12 becomes 41559. Find 41559 and replace with ‘10-12
  • 12-14 becomes 41622. Find 41622 and replace with ‘12-14

Following is a summary of the workflow:

  1. Import an existing CSV that contains data.
  2. Change the Excel column to display as text.
  3. Find and replace all instances of 41338, 41402, 41559, and 41622. Replace with their corresponding age range, but include a leading apostrophe before the age range in the Find/Replace dialog box to force Excel to write the age range as a string.
  4. Continue with the workflow.
  5. The above task can be automated and expedited by creating a macro.

How Tagger Behaves

The following are specific behaviors of Tagger that will help the user manage his or her workflow:

  • The user must first create a new resource by clicking on the “New” button before the user can add information to any field.
  • Tagger can import many records, but the exact number is limited to the amount of memory available on the user’s computer and the timeout setting of the browser. During testing, 500 records were imported in about 30 seconds. The user may be able to import more or less, depending on a variety of factors specific to the user’s machine.
  • If the user works on items without signing into an inBloom account, the user must export those items before signing in with the user’s inBloom account and then re-import the items after signing in. Otherwise, the user will lose his or her work after signing in with an inBloom account.
  • After the user signs in, the user can recall any item he or she has previously published to the Learning Registry by clicking on the “History” button. Then, the user can make changes and resubmit or export. Currently, the Learning Registry does not have the ability to update records, so it will accept an updated record while marking the original record to be ignored.
  • The user can export and then re-import any number of items. The data should be the same on import as it was at export.

Reporting Bugs and/or Features

The user can report bugs by clicking on the “Report Issue” button on the upper right of the screen. The report issue form asks the user a number of questions concerning the bug and the user’s system. Though only a few items are required, providing as much information as possible to recreate the bug will help the support team identify and fix the issue.
